# Takos Deploy v1.0 Implementation Checklist

Use this checklist to verify that an implementation conforms to the v1.0 Core
semantics defined in
[`core/01-core-contract-v1.0.md`](./core/01-core-contract-v1.0.md). The 15
invariants listed in spec § 2 are the load-bearing semantics; this checklist
follows them and adds the boundary / packaging items the migration re-validates.

> Implementation status: see [Current Implementation Note](./current-state).

- [x] Core contract uses MUST / SHOULD / MAY consistently.
- [x] TypeScript model and JSON Schema match the normative record names
      (`Deployment`, `ProviderObservation`, `GroupHead`).
- [x] Descriptor resolution policy decisions are recorded under
      `Deployment.policy_decisions[]`.
- [x] Descriptors are declarative and cannot hide executable provider logic.
- [x] Provider capabilities constrain or reject contracts but do not reinterpret
      them.
- [x] Deployment.desired immutability after `applied` is covered by conformance
      tests (invariant 11).
- [x] `Deployment.desired.activation_envelope` records the desired routed
      serving envelope (invariant 13).
- [x] GroupHead.current_deployment_id selects the current applied Deployment and
      advances atomically (invariant 14).
- [x] Route-specific assignment, if implemented, is stored in
      `Deployment.desired.activation_envelope.route_assignments`.
- [x] Publication does not imply injection (invariant 4).
- [x] Resource credentials are not publication outputs by default (invariant 8).
- [x] ProviderObservation is observed-side, never canonical (invariant 12).
- [x] Rollback uses the retained `Deployment.input.manifest_snapshot` plus the
      Shape/provider metadata snapshot (or compatibility `descriptor_closure`)
      of the target Deployment, and is a `GroupHead` pointer move rather than a
      new Deployment (invariant 15).
- [x] Current Takos migration plan is mapped to the Deployment-centric Core
      records.
- [x] Official descriptor set is available before Deployment resolution.
- [x] Condition reason catalog is used consistently by PaaS API/controllers and
      exported through the contract/OpenAPI consumed by CLI and UI.
- [x] Production self-hosted plugin implementation is outside the kernel; the
      kernel exposes the required ports, selection map, and trusted registration
      path for an operator-provided bundle.
- [x] Third-party production plugin enablement uses the trusted signed-manifest
      installer registered against the operator-controlled trusted registry;
      dynamic module loading is local/operator-only (see
      [Current Implementation Note](./current-state) for detail).
- [x] Production cloud provider plugin implementations are outside the PaaS
      kernel; Cloudflare, AWS, GCP, Kubernetes, Neon, R2, S3, and similar
      descriptor-backed targets are selected only when an operator registers an
      external bundle that declares the required ports and fails closed on its
      own missing provider/client references.
- [x] External-boundary resource access path policy validation blocks the
      Deployment unless `Deployment.desired.runtime_network_policy` explicitly
      allows external egress.

## Cross-instance service binding foundation (v1.x scope)

下記項目は v1.0 conformance には含まれませんが、mainline には cross-instance
service binding の consumer-side foundation が入り始めています。 provider
publish / cache refresh / durable audit まで含む full conformance は継続 work
です。

- [x] Invariant 16 foundation: public deploy route resolves `imports[]` through
      `serviceResolvers[]`, fetches provider-signed `ServiceDescriptor`,
      verifies Ed25519 signature with the pinned `publicKey`, checks contract /
      expiry, and rejects apply before provider execution on failure.
- [ ] Invariant 16 durable audit: all verification attempts (including failure)
      are append-only audit records.
- [x] Invariant 17 foundation: resolved descriptor digest / resolver / provider
      instance / expiry are pinned into
      `ManifestResource.metadata.takosumiServiceImports`; `from.import` bindings
      keep `alias/endpointRole/field` identity in the deploy graph.
- [ ] Invariant 17 full closure: resolved `ServiceDescriptor` records are
      durably pinned in the Deployment's service import metadata (and in
      compatibility descriptor projections when emitted); refresh requires a new
      `Deployment`.
- [x] Invariant 18: kernel SHALL NOT operate a service registry。 service id
      から `ServiceDescriptor` への resolution は anchor (operator-injected via
      manifest `serviceResolvers[]`) に delegate。

詳細は
[`../architecture/cross-instance-service-binding`](../architecture/cross-instance-service-binding)
と
[`../reference/service-identifier-spec`](../reference/service-identifier-spec)
を参照。
