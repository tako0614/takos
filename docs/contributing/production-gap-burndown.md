This is the concise burndown for the Takosumi kernel release surface that the
Takos product shell depends on. It records the strongest state currently present
across the `takos/` product root and sibling `../takosumi/` kernel repo, and
keeps external self-host/cloud proof separate from the local release gate.
Unless the command starts with `cd ../takosumi`, run proof commands from
`/home/tako/Desktop/takos/takos`.

## Current validation baseline

- Deployment-centric kernel surface is implemented in
  `../takosumi/packages/kernel/src/domains/deploy/deployment_service.ts`.
- Current takosumi kernel deploy surface is `POST /v1/deployments` with a
  compiled Shape manifest. The older `/api/public/v1/deployments` family is
  migration compatibility surface and must not be described as the current
  kernel public contract.
- `scripts/release-gate.ts` is the safe local Takos product gate. It runs check,
  docs build, service-set and architecture validators, distribution /
  observability / patch / migration / legal / release-promotion validators, Helm
  and Terraform policy checks, release manifest generation, and local config
  validation.
- Official distribution manifests live in `deploy/distributions`. They are
  public templates, not live proof. Live self-host/cloud proof requires
  operator-owned infrastructure, credentials, and external plugin live tasks.

## Remaining high-priority gaps

- Live self-host/cloud proofs are not covered by local kernel gates. Keep them
  as operator-owned evidence attached to the distribution target.
- RepairPlan rematerialization and rollback-with-migration resource semantics
  remain backlog unless covered by a concrete test in a later turn.
- Canary side-effect safety still needs tests for queue DataContract policy and
  hard-fail shadow traffic side-effect semantics.
- ChangeSetPlan partial-failure semantics remain an acceptance gap.
